# Phase 02.1 Plan 01: Remediate CR-01 cancel-on-timeout and CR-02 client_order_id persistence Summary

**EXEC-04 idempotency invariant restored on the write path (Kalshi client_order_id, not server order_id, is now persisted to execution_orders.client_order_id) and EXEC-05 timeout-recovery rewritten to use list_open_orders_by_client_id(prefix) so orphaned Kalshi orders from network blips during place_fok are actually cancelled instead of leaked as live resting orders.**

## Accomplishments

- Added `Order.external_client_order_id: Optional[str] = None` field; serialized in both `to_dict` and `to_audit_dict`.
- KalshiAdapter `place_fok` success path populates `external_client_order_id` from the engine-chosen ARB-prefixed string (e.g. `ARB-000042-YES-deadbeef`); KalshiAdapter `_order_data_to_order` surfaces the platform's `client_order_id` field into the same Order field so list_open_orders_by_client_id-returned Orders carry the key for end-to-end recovery threading.
- PolymarketAdapter all three Order(...) constructors (`_order_from_response`, `_order_from_record`, `_failed_order`) explicitly set `external_client_order_id=None` for parity and to document the intentional omission.
- `ExecutionEngine._derive_client_order_id` is now a one-line `return order.external_client_order_id` — the buggy `-` heuristic on `order.order_id` (which after Kalshi success holds the platform-assigned id) is gone.
- `ExecutionEngine._place_order_for_leg` `asyncio.TimeoutError` branch rewritten to: compute `prefix = f"{arb_id}-{SIDE}-"`, `await adapter.list_open_orders_by_client_id(prefix)` (try/except + log), iterate and `await adapter.cancel_order(real)` on each (per-call try/except + log), set final status based on cancelled-any / found-but-cancel-failed / no-match. The synthetic `partial.order_id` is now used only as the DB row PK; the real order(s) returned from the lookup are the only objects passed to `cancel_order`.
- Engine threads `external_client_order_id` from the first matched real order into the synthetic partial so the persisted DB row carries the real idempotency key on the timeout-CANCELLED path.
- 10 new/updated regression tests (8 engine + 1 Kalshi adapter + 1 Polymarket adapter), plus 2 file-level test name aliases for VALIDATION.md row-name compatibility — all 12 pass.

## Decisions Made

1. **Aliasing test names instead of renaming** — VALIDATION.md and PATTERNS.md cite different test names for the same intent (e.g. `test_place_fok_populates_external_client_order_id` vs `test_place_fok_returns_external_client_order_id`). Implemented under PATTERNS.md naming and added module-level alias assignments so both names resolve to the same test. Net: 12 pytest node IDs, 10 unique implementations.
2. **Polymarket explicit `=None` (Option B from PATTERNS.md)** — Documents intent at the call site. No functional change but improves auditability and prevents future regressions where someone might add the field without thinking about Polymarket parity.
3. **Threading `external_client_order_id` from matched real order into synthetic partial in the timeout branch** — Per the `test_timeout_recovery_end_to_end` invariant: when timeout-recovery cancels real orphaned orders, the persisted DB row must carry the real idempotency key, not NULL. Without this, the timeout-CANCELLED path would lose the linkage between the partial DB row and the real Kalshi orders we just cancelled.
4. **Populating `external_client_order_id` in Kalshi `_order_data_to_order`** — Without this, the production list_open_orders_by_client_id flow would return Orders with `external_client_order_id=None`, breaking the threading described in (3) end-to-end. The mock-based test `test_timeout_recovery_end_to_end` happened to set the field directly on the mock real order, but production needed the adapter to actually populate it from the Kalshi API response.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Threading external_client_order_id from matched real order into synthetic partial Order in the timeout branch**

- **Found during:** Task 3 (CR-01 fix verification — `test_timeout_recovery_end_to_end` failed)
- **Issue:** After implementing the CR-01 timeout-recovery loop verbatim per PLAN.md Action block, the `test_timeout_recovery_end_to_end` test failed with `assert None == 'ARB-42-YES-abcd1234'`. The plan-prescribed code persisted the synthetic `partial` Order via `engine.store.upsert_order(...)`, but `partial.external_client_order_id` was None (because `partial` is constructed without it). The test (and the must-have invariant) require the persisted client_order_id to be the prefix-derived ARB string from the matched real order.
- **Fix:** Inside the timeout branch, after successfully retrieving `open_orders`, iterate once to find the first real order with non-None `external_client_order_id` and copy it onto `partial.external_client_order_id`. This makes `_derive_client_order_id(partial)` return the real key and `engine.store.upsert_order(client_order_id=...)` receive the correct value.
- **Files modified:** arbiter/execution/engine.py
- **Verification:** `pytest arbiter/execution/test_engine.py::test_timeout_recovery_end_to_end -q` passes; full suite still green.
- **Committed in:** 7d769af (Task 3 commit)

**2. [Rule 2 — Missing critical functionality] KalshiAdapter `_order_data_to_order` did not populate `external_client_order_id` from the Kalshi API response**

- **Found during:** Task 3 (analyzing why Deviation 1 was needed end-to-end in production)
- **Issue:** The mock test `test_timeout_recovery_end_to_end` injects a real order with `external_client_order_id` already set, but in production, `KalshiAdapter._order_data_to_order` (the function that converts the Kalshi `/portfolio/orders?status=resting` response into Order objects) was not extracting the `client_order_id` field from the API response into the new dataclass field. Without this, real production list_open_orders_by_client_id calls would return Orders with `external_client_order_id=None`, defeating Deviation 1's threading even when the engine logic was correct.
- **Fix:** Read `od.get("client_order_id")` and pass it as `external_client_order_id=str(cid) if cid else None` on the returned Order. Surfaces the engine-chosen key from the platform response so the engine recovery path can thread it back into ExecutionStore.
- **Files modified:** arbiter/execution/adapters/kalshi.py
- **Verification:** Production code path now matches mock-test expectation. Full suite still green.
- **Committed in:** 7d769af (Task 3 commit, alongside CR-01 fix in same file)
