# Phase 02.1: Code Review Report

**Reviewed:** 2026-04-16
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Phase 02.1 remediates two critical issues from the Phase 2 review:

- **CR-01** (timeout-cancel): On `asyncio.wait_for` timeout in `_place_order_for_leg`, the engine now queries `adapter.list_open_orders_by_client_id(prefix)` to discover real platform orders by `client_order_id` prefix, then cancels each match. Previously it called `adapter.cancel_order(partial)` with a synthetic in-memory `Order` whose `order_id` was a non-existent string, guaranteed to 404 on Kalshi.
- **CR-02** (`client_order_id` persistence): A new `Order.external_client_order_id: Optional[str]` field carries the engine-chosen `ARB-{n}-{SIDE}-{hex}` idempotency key from the adapter back to the engine. `_derive_client_order_id` now reads this field instead of the unsound `"-" in order.order_id` heuristic, so the DB column holds the real idempotency key rather than Kalshi's server-assigned `order_id`.

The remediation is correct and well-tested for the single-order recovery path. The Kalshi and Polymarket adapter changes are minimal and consistent (Polymarket explicitly opts out by setting `external_client_order_id=None` everywhere). No security issues, no resource leaks, no boundary violations were introduced.

Two warnings cover semantic correctness gaps in the multi-order timeout-recovery code path; four info items cover test duplication and minor message-clarity improvements.

## Warnings

### WR-01: Multi-order timeout recovery reports CANCELLED even when only one of several cancels succeeds

**File:** `arbiter/execution/engine.py:797-808`
**Issue:** The recovery loop sets `cancelled_any = True` on the first successful `adapter.cancel_order(real)` call, then unconditionally sets `partial.status = OrderStatus.CANCELLED` if `cancelled_any` is true — regardless of how many of the other open orders failed to cancel. The accompanying error string says `f"cancelled {found_count} orphaned order(s)"`, but `found_count` is the total number of orders found, not the number actually cancelled. In the (admittedly rare) case where `list_open_orders_by_client_id` returns more than one match (e.g. a transient duplicated submit) and only some cancels succeed, the persisted state will incorrectly report all orders as cancelled while live orders remain on the platform. This violates the safety-over-speed invariant in `CLAUDE.md` ("cannot afford to lose capital to bugs").

**Fix:** Track per-order outcomes and gate the CANCELLED transition on full success. Suggested patch:

```python
cancelled_count = 0
failed_count = 0
for real in open_orders:
    try:
        if await adapter.cancel_order(real):
            cancelled_count += 1
        else:
            failed_count += 1
    except Exception as cancel_exc:
        failed_count += 1
        logger.warning(
            "timeout_recovery.cancel_raised platform=%s order_id=%s err=%s",
            platform, real.order_id, cancel_exc,
        )

if found_count == 0:
    partial.status = OrderStatus.FAILED
    partial.error += "; no matching open order found - platform may have rejected or never received"
elif cancelled_count == found_count:
    partial.status = OrderStatus.CANCELLED
    partial.error += f"; cancelled {cancelled_count} orphaned order(s) found by client_order_id prefix"
else:
    # Partial cancel — at least one live order remains. Treat as FAILED so the
    # operator-facing surface flags this for manual reconciliation.
    partial.status = OrderStatus.FAILED
    partial.error += (
        f"; cancelled {cancelled_count}/{found_count} orphaned order(s)"
        f" — {failed_count} still live on platform, manual reconciliation required"
    )
```

### WR-02: Only the first non-empty `external_client_order_id` is propagated into the persisted partial

**File:** `arbiter/execution/engine.py:784-787`
**Issue:** When the timeout-recovery lookup returns multiple open orders, the loop:
```python
for real in open_orders:
    if real.external_client_order_id:
        partial.external_client_order_id = real.external_client_order_id
        break
```
takes the first non-empty `external_client_order_id` and writes it onto the synthetic `partial`. Because the partial's `client_order_id` is what gets persisted via `ExecutionStore.upsert_order(client_order_id=...)`, only one of the (potentially multiple) real orders is recorded against this DB row. The other live orders’ idempotency keys are silently dropped — and in CR-02's intended audit story, `client_order_id` is supposed to be the canonical idempotency key for a given (arb, side) pair. In practice today the prefix is `ARB-{n}-{SIDE}-` so all matches share the same arb/side root, but the per-order suffix (`-{hex}`) differs — meaning two real orders for the same prefix would collapse to a single recorded `client_order_id`, hiding the second order entirely from the persistence layer.

**Fix:** This pairs with WR-01. If you adopt the partial-recovery semantics suggested there, the synthetic `partial` should not be the only persisted artifact when `found_count > 1`. Either:
- Persist each real order separately via `self.store.upsert_order(real, arb_id=arb_id, client_order_id=real.external_client_order_id)` before/after attempting cancel, or
- Record an explicit incident (severity=critical) when `found_count > 1` listing every real order’s `(order_id, client_order_id, status)` so operators can reconcile manually.

Concretely:

```python
# After the lookup, before the cancel loop:
if self.store is not None:
    for real in open_orders:
        try:
            await self.store.upsert_order(
                real, arb_id=arb_id,
                client_order_id=real.external_client_order_id,
            )
        except Exception as exc:
            logger.warning("ExecutionStore upsert_order (orphan) failed: %s", exc)

if found_count > 1:
    await self.record_incident(
        arb_id=arb_id, canonical_id=canonical_id,
        severity="critical",
        message=f"Timeout recovery found {found_count} open orders for prefix {prefix}",
        metadata={"orders": [o.to_dict() for o in open_orders]},
    )
```

## Info

### IN-01: Two timeout-recovery tests are byte-for-byte equivalent

**File:** `arbiter/execution/test_engine.py:651-683` and `arbiter/execution/test_engine.py:686-718`
**Issue:** `test_timeout_looks_up_real_orders_by_client_id` and `test_timeout_uses_list_open_orders_by_client_id_with_correct_prefix` set up identical mocks (hanging `place_fok`, empty lookup return, cancel returns `False`), call `_place_order_for_leg` with `arb_id="ARB-42"`/`side="yes"`, and make the same three assertions:
```python
adapter.list_open_orders_by_client_id.assert_awaited_once_with("ARB-42-YES-")
assert order.status == OrderStatus.FAILED
assert "no matching open order found" in order.error
adapter.cancel_order.assert_not_called()
```
Only the docstrings differ. Running both adds runtime cost (each does a real `asyncio.wait_for` + 0.1s timeout) without adding coverage.

**Fix:** Delete one of the two (suggest deleting `test_timeout_uses_list_open_orders_by_client_id_with_correct_prefix` since the prefix-shape assertion is already covered by the surviving `test_timeout_looks_up_real_orders_by_client_id`). If the VALIDATION.md row mapping requires a second name, leave a one-line alias instead, mirroring the pattern already used at the bottom of `test_kalshi_adapter.py:391` (`test_place_fok_populates_external_client_order_id = test_place_fok_returns_external_client_order_id`).

### IN-02: Error string for the no-match path implies the platform may still hold a resting order

**File:** `arbiter/execution/engine.py:809-814`
**Issue:** When `list_open_orders_by_client_id` returns `[]`, the partial's error becomes:
```
local timeout after 10.0s; no matching open order found - platform may have rejected or never received
```
This is technically correct for Kalshi (whose `/portfolio/orders?status=resting` is the source of truth for open orders), but on Polymarket the adapter's `list_open_orders_by_client_id` *always* returns `[]` (it has no `client_order_id` concept and intentionally degrades — see `polymarket.py:488-503`). Operators reading this error on a Polymarket leg would conclude the order never reached the platform, when in fact the engine simply has no way to verify either way. This is an operational footgun, not a bug.

**Fix:** Branch the error message on platform, or call `adapter.list_open_orders_by_client_id` only when the adapter advertises support. Lightweight option:

```python
if found_count == 0:
    if platform == "polymarket":
        partial.error += (
            "; lookup-by-client-id unsupported on this platform"
            " - manual reconciliation required"
        )
    else:
        partial.error += (
            "; no matching open order found"
            " - platform may have rejected or never received"
        )
```
A heavier-but-cleaner option is to add `supports_client_order_id_lookup: bool` to the `PlatformAdapter` Protocol and drive recovery semantics from it.

### IN-03: `external_client_order_id` is appended to `to_dict()` and `to_audit_dict()` without an API/DB schema contract note

**File:** `arbiter/execution/engine.py:63-95`
**Issue:** Both `Order.to_dict()` and `Order.to_audit_dict()` now include `external_client_order_id` unconditionally. `to_dict` flows out through `ArbExecution.to_dict` → `_publish_execution` → WebSocket subscribers (`api.py`), and through any HTTP endpoints that serialize `ArbExecution`. Downstream consumers (the dashboard, Postgres `execution_orders` row mirror, JSON trade log) will silently start receiving the new field. There is no schema-version bump or a doc note, and the dashboard JS in `arbiter/web/dashboard.js` may have schema-validation that rejects extra fields. (Did not verify the dashboard side — flagging as a release-note item.)

**Fix:** Add a short comment on the field documenting that it's part of the public `to_dict` contract, and either:
- audit `arbiter/web/dashboard.js` for strict schema validation, or
- gate the field on a non-None value (`if self.external_client_order_id is not None: out["external_client_order_id"] = ...`) to avoid changing the wire shape for the common Polymarket path where it’s always `None`.

### IN-04: `_derive_client_order_id` now ignores `order.platform` — comment is slightly stale

**File:** `arbiter/execution/engine.py:829-842`
**Issue:** The new implementation simply returns `order.external_client_order_id`, regardless of platform. The docstring still motivates the change in terms of Kalshi/Polymarket distinction ("Polymarket has no client_order_id concept and leaves the field None"), which is accurate but slightly misleading — the helper is now platform-agnostic by construction; the platform-awareness is enforced by the adapters when they populate the field. This is a documentation polish, not a correctness issue.

**Fix:** Trim the docstring to:
```python
"""Return the adapter-populated client_order_id, or None.

The adapter is responsible for populating ``Order.external_client_order_id``
with the engine-chosen idempotency key on platforms that support one
(currently Kalshi). Adapters on platforms without a client_order_id concept
(Polymarket) leave the field as None.

Replaces the prior ``"-" in order.order_id`` heuristic, which was unsound
because Kalshi's server-assigned ``order_id`` values also contain ``-``,
causing the DB ``client_order_id`` column to be populated with the platform
id rather than the engine-chosen idempotency key (CR-02).
"""
```
