## Summary

Phase 02.1 remediates two critical issues surfaced in the Phase 2 code review (`02-REVIEW.md`) that block live Kalshi trading. Both are narrowly scoped in-process bug fixes — no new dependencies, no schema changes, no wiring changes in `main.py`. The entire fix lives in three files: `arbiter/execution/engine.py` (both bugs), `arbiter/execution/adapters/kalshi.py` (one new field population + one bug-adjacent log/error-code tidy), and `arbiter/execution/adapters/polymarket.py` (parity update for the new `Order` field). Test additions land in `arbiter/execution/test_engine.py`, `arbiter/execution/adapters/test_kalshi_adapter.py`, and optionally `arbiter/execution/adapters/test_polymarket_adapter.py`.

**Why both should land together:** CR-01's fix (timeout-recovery via `list_open_orders_by_client_id`) only works if CR-02 is also fixed — otherwise the `client_order_id` column in the DB is populated with Kalshi's server-side order_id, and `list_open_orders_by_client_id(prefix="ARB-000042-YES-")` never matches anything. Splitting into two plans risks landing CR-01 on top of CR-02's broken state and shipping a half-fix. They are **functionally coupled** and **localizable to a single small plan**.

**Primary recommendation:** One plan, one wave, two tasks. Task 1: add `external_client_order_id: Optional[str] = None` field to the `Order` dataclass and populate it in `KalshiAdapter.place_fok` (CR-02 Option A from REVIEW.md, explicitly preferred by VERIFICATION.md). Task 2: rewrite the `asyncio.TimeoutError` branch of `engine._place_order_for_leg` to use `list_open_orders_by_client_id(prefix=f"{arb_id}-{side.upper()}-")` + cancel-each-match, and tighten `_derive_client_order_id` to return `order.external_client_order_id` (CR-01). Both changes are surgical (<100 LOC total, tests included). No migration needed — the DB schema is unchanged, and retroactive `client_order_id` values in existing rows are either `NULL` (pre-upsert) or wrong-but-harmless (CR-02 was always write-side broken; no consumer depends on the current value yet).

### Pattern 1: Dataclass evolution without migration

**What:** Adding an optional field with a default value to a dataclass is backward-compatible — existing callers don't break, and no DB migration is needed because the new field is not persisted (it's an in-process-only field; the already-persisted `client_order_id` *column* remains the system of record).

**When to use:** The field is an in-process carrier of data generated in one layer and consumed in another layer of the same process. It never needs to survive a restart.

**Example (CR-02 fix):**
```python
# arbiter/execution/engine.py — Order dataclass
@dataclass
class Order:
    order_id: str
    platform: str
    market_id: str
    canonical_id: str
    side: str
    price: float
    quantity: int
    status: OrderStatus
    fill_price: float = 0.0
    fill_qty: int = 0
    timestamp: float = 0.0
    error: str = ""
    external_client_order_id: Optional[str] = None   # ← CR-02 fix: carries the Kalshi client_order_id from adapter to engine
```

**Key property:** Because the field has a default of `None` and is placed at the end (after other defaulted fields), all positional constructors continue to work. `Order(order_id="x", platform="kalshi", ...)` calls do not need to be rewritten. [VERIFIED: grep across arbiter/ shows no positional Order() calls using the last field]

### Pattern 2: Adapter-owned secret → engine-visible metadata

**What:** The Kalshi `client_order_id` is generated inside `KalshiAdapter.place_fok` (format: `f"{arb_id}-{side.upper()}-{uuid.uuid4().hex[:8]}"`) — the engine never sees it. To persist it, we carry it back on the returned `Order` object. The engine reads `order.external_client_order_id` and threads it into `store.upsert_order(client_order_id=...)`.

**When to use:** Adapter internals need to be visible to the engine's persistence path without leaking into every call site or breaking the adapter seam.

**Example (CR-02 fix):**
```python
# arbiter/execution/adapters/kalshi.py — adjust return in place_fok
# Current (kalshi.py:174):
return Order(
    order_id=str(order_data.get("order_id", client_order_id)),
    platform="kalshi",
    # ...
    timestamp=now,
)

# Fixed:
return Order(
    order_id=str(order_data.get("order_id", client_order_id)),
    platform="kalshi",
    # ...
    timestamp=now,
    external_client_order_id=client_order_id,   # ← CR-02: preserve for engine's persistence path
)
```

And **also** in every error path inside `place_fok` that generates a client_order_id: the `_failed_order` helper currently returns `Order(order_id=f"{arb_id}-{side.upper()}-KALSHI", ...)` — since the failed order was never submitted to Kalshi (or rejected by it), there is no `client_order_id` to persist. Leave `external_client_order_id=None` in the failure path so `_derive_client_order_id` returns None and the DB stores NULL (which matches the existing partial-index predicate `WHERE client_order_id IS NOT NULL`). [VERIFIED: sql/migrations/001_execution_persistence.sql:44]

### Pattern 3: Unknown-state fan-out cancel

**What:** When a timeout fires, the engine does not know whether (a) the request never reached Kalshi, (b) the request reached Kalshi but the response got lost, or (c) the request succeeded and created a resting order. Case (c) is the dangerous one — a leaked resting order can fill later, creating unhedged risk. The fix assumes case (c) is possible and queries the platform for any orders matching the arb's client_order_id prefix. If zero are found, case (a) or (b) — nothing to cancel. If ≥1 are found, case (c) — cancel each.

**When to use:** Any best-effort recovery where the local process lost visibility but the remote system may still hold state. This is the standard pattern for at-least-once → at-most-once conversion via platform-side introspection.

**Example (CR-01 fix):**
```python
# arbiter/execution/engine.py — rewrite asyncio.TimeoutError branch of _place_order_for_leg
except asyncio.TimeoutError:
    partial = Order(
        order_id=f"{arb_id}-{side.upper()}-{platform.upper()}",  # unchanged — this is the DB PK for the failed row
        platform=platform,
        market_id=market_id,
        canonical_id=canonical_id,
        side=side,
        price=price,
        quantity=qty,
        status=OrderStatus.PENDING,
        timestamp=time.time(),
        error=f"local timeout after {self.execution_timeout_s}s",
    )

    cancelled_any = False
    found_any = False
    prefix = f"{arb_id}-{side.upper()}-"

    try:
        open_orders = await adapter.list_open_orders_by_client_id(prefix)
    except Exception as exc:
        logger.warning(
            "timeout_recovery.lookup_failed",
            extra={"arb_id": arb_id, "platform": platform, "err": str(exc)},
        )
        open_orders = []

    for real_order in open_orders:
        found_any = True
        try:
            if await adapter.cancel_order(real_order):
                cancelled_any = True
        except Exception as cancel_exc:
            logger.warning(
                "timeout_recovery.cancel_raised",
                extra={"arb_id": arb_id, "platform": platform,
                       "order_id": real_order.order_id, "err": str(cancel_exc)},
            )

    if cancelled_any:
        partial.status = OrderStatus.CANCELLED
        partial.error += f"; cancelled {cancelled_any}-of-{len(open_orders)} orphaned orders"
    elif found_any:
        # we found orders but failed to cancel them — WORST CASE
        partial.status = OrderStatus.FAILED
        partial.error += f"; found {len(open_orders)} orphaned orders but cancel failed — manual reconciliation required"
    else:
        partial.status = OrderStatus.FAILED
        partial.error += "; no matching open order found - platform may have rejected or never received"
    order = partial
```

**Why this is still best-effort:** Between `list_open_orders_by_client_id` and `cancel_order`, the order could fill. Kalshi's DELETE on a just-filled order returns 4xx (order already terminal), which our `cancel_order` implementation reports as False. In that case we land in the "found but cancel failed" branch with an accurate error — the resting leak has converted into a filled leg that the engine will pick up via `recovery.reconcile_non_terminal_orders` on the next restart or via the audit path. Document this edge case in the plan's threat model.

### Anti-Patterns to Avoid

- **Passing synthetic order_ids to `cancel_order`:** The original CR-01 bug. Adapters are HTTP clients; `DELETE /portfolio/orders/{id}` with `{id}` = `ARB-000042-YES-KALSHI` returns 404 every time. The engine must look up real platform order_ids before calling cancel.
- **Storing platform ids in `client_order_id` columns:** The original CR-02 bug. The `client_order_id` column's entire purpose is idempotency — a stable string *the client chose* before submission. Storing the server-assigned id defeats `list_open_orders_by_client_id` (which filters by prefix match on the client-chosen string).
- **Making the DB column's uniqueness assumption:** `idx_execution_orders_client_id` is a NON-UNIQUE index (regular `CREATE INDEX ... ON execution_orders(client_order_id)` — not `CREATE UNIQUE INDEX`). REVIEW.md misstates this as "partial unique index" — it is not unique. [VERIFIED: arbiter/sql/migrations/001_execution_persistence.sql:42-44 shows plain `CREATE INDEX IF NOT EXISTS`]. The planner should NOT rely on DB-level uniqueness to catch duplicate client_order_ids — uniqueness lives at application level via Kalshi's own idempotency semantics.
- **Generating new client_order_ids in the timeout-recovery path:** The timeout branch's job is to CLEAN UP, not submit. Do not generate a new `client_order_id` during recovery. The recovery path never calls `place_fok`.

## Common Pitfalls

### Pitfall 1: The existing `test_engine_timeout_triggers_cancel` test is a false green

**What goes wrong:** The test at `arbiter/execution/test_engine.py:510-538` passes today despite CR-01 being live — it mocks `cancel_order = AsyncMock(return_value=True)` and only asserts `cancel_order.assert_awaited_once()`. It does NOT assert WHICH order was passed to cancel_order. So the synthetic-order_id bug slips through.

**Why it happens:** Mock tests assert "X was called" without asserting "X was called with Y." When the adapter is real (integration test against Kalshi), the bug manifests as a 404 — but the unit test never sees that.

**How to avoid:** The new tests for CR-01 MUST assert:
1. `list_open_orders_by_client_id` was called with the correct prefix (`f"{arb_id}-{side.upper()}-"`)
2. `cancel_order` was called with each order returned by `list_open_orders_by_client_id` — not with the synthetic Order
3. When `list_open_orders_by_client_id` returns `[]`, `cancel_order` is NOT called, AND the final Order.status is FAILED with error containing "no matching"
4. When `cancel_order` succeeds for at least one order, final Order.status is CANCELLED
5. When `list_open_orders_by_client_id` raises, the engine degrades gracefully (no raise across `_place_order_for_leg` boundary)

**Warning signs:** If a test passes but doesn't inspect `.call_args`, the test may be exercising the wrong invariant. Grep for `.assert_called` without nearby `.call_args` / `.assert_called_with(...)`.

### Pitfall 2: `_derive_client_order_id` currently returns `order.order_id` for ANY Kalshi order with a `-`

**What goes wrong:** The current logic at `engine.py:790` is `if order.platform == "kalshi" and order.order_id and "-" in order.order_id: return order.order_id`. After the Kalshi adapter overwrites `order.order_id` with the server-assigned id (which also often contains `-`), this function returns the wrong string.

**Why it happens:** The `-` heuristic was intended to detect the `ARB-xxx-YES-hex` format, but Kalshi's server-assigned order_ids can also contain `-` (they're UUID-like strings). The heuristic is too permissive.

**How to avoid:** Replace the heuristic with an explicit field read: `return order.external_client_order_id`. If the adapter didn't populate the field, the function returns None — which is correct for both Polymarket (no client_order_id concept) and Kalshi failure paths (no client_order_id to persist).

**Warning signs:** Any `order_id`-string heuristic that tries to distinguish "engine-chosen ID" from "platform-assigned ID" after both have been written to the same field. Use separate fields instead.

### Pitfall 3: Kalshi's `/portfolio/orders` returns paginated results, but `_list_orders` ignores pagination

**What goes wrong:** Kalshi's `/portfolio/orders` accepts `limit` (1-1000, default 100) and `cursor` for pagination [VERIFIED: docs.kalshi.com/api-reference/orders/get-orders]. The current `KalshiAdapter._list_orders` at kalshi.py:354 does NOT send a `limit`, does NOT follow the `cursor`. For accounts with ≤100 open orders this is fine; for larger accounts, the lookup could miss the target order.

**Why it happens:** Phase 2 wrote the lookup assuming small accounts (capital constraint in CLAUDE.md is "under $1K per platform initially"). At $1K initial capital with $10-50 per trade, max concurrent orders is <20. But the lookup should be robust for Phase 4.

**How to avoid:** The CR-01 fix SHOULD pass an explicit `limit` (e.g. 1000) to the Kalshi query and, if the response includes a `cursor`, page through subsequent requests until cursor is null or the target prefix is found. For Phase 02.1 scope, a limit=1000 single-call (no pagination) is ACCEPTABLE — document as a known limitation and a Phase 4 tightening item. Do NOT extend scope to full pagination; it would triple the test surface.

**Warning signs:** API endpoints that return a `cursor` field in the response but code that never inspects it. The result looks correct in small-scale tests but silently truncates in production.

### Pitfall 4: `list_open_orders_by_client_id` may return slightly-stale data

**What goes wrong:** The adapter reads `status=resting` orders from Kalshi. If an order was placed but status is still `pending` on Kalshi's side (not yet resting), it won't be in the result. The timeout fired because Kalshi was slow — so Kalshi-side slowness is exactly the scenario where the order may still be `pending`.

**Why it happens:** Kalshi order lifecycle: pending → resting → (executed | canceled). `pending` is transient and usually sub-second, but under a 10s local timeout the order could be stuck in `pending` if Kalshi itself is backed up.

**How to avoid:** Extend `list_open_orders_by_client_id` to query BOTH `status=resting` AND `status=pending` (two separate calls, then merge). OR: widen the query to query all non-terminal states by making multiple calls. For Phase 02.1 scope, query `resting` first (matches current adapter behavior), and document the `pending` gap as a Phase 4 refinement. Low priority because Kalshi `pending` rarely exceeds a few seconds and the overall recovery is best-effort.

**Warning signs:** Recovery logic that only queries one state of a multi-state lifecycle. If the lifecycle has N non-terminal states, recovery should check at least the two that cover >99% of stuck orders.

### Pitfall 5: Existing DB rows with wrong `client_order_id` values

**What goes wrong:** Any `execution_orders` row written by Phase 2 during dry-run or testing has `client_order_id` = Kalshi's platform id (wrong per CR-02). If a future query does `SELECT * FROM execution_orders WHERE client_order_id LIKE 'ARB-%'` (say, during audit), those rows won't appear — they look like they have no client_order_id.

**Why it happens:** CR-02 was a write-side bug. Existing rows carry the bug's legacy.

**How to avoid:** The planner has two options:
- **Option A (preferred):** Do nothing. Phase 2 has not gone to live trading. Any test/dry-run rows in the DB are not production data. They will be overwritten on `ON CONFLICT DO UPDATE` if the same `order_id` is upserted again, or they remain as historical noise that does no harm.
- **Option B:** Include a one-shot SQL statement in the plan: `UPDATE execution_orders SET client_order_id = NULL WHERE platform = 'kalshi' AND client_order_id NOT LIKE 'ARB-%';`. This clears the wrong values. Safe because the column is nullable and non-unique-indexed.

Document both options in the plan. Recommend Option A unless the operator has specific audit concerns.

**Warning signs:** Any write-side bug that affected rows still present in the DB. Before writing a migration, verify the read paths actually care about the affected data.

### Pitfall 6: Tenacity retry inside `place_fok` re-uses the same client_order_id; engine's recovery path generates NONE

**What goes wrong:** `KalshiAdapter._post_order` is `@transient_retry()`-decorated. If a transient network error occurs, tenacity retries the POST with the SAME `client_order_id` — safe because Kalshi dedupes on that key. But the engine's CR-01 fix does NOT need to generate a new `client_order_id` on recovery — recovery is a CLEANUP path, not a resubmit path. If a future change adds a "retry after timeout" behavior at the engine level, the engine would need to either reuse the original `client_order_id` (find via the partial order, or pass it down) OR generate a new one and accept that the orphaned one will be recovered later. Current engine has NO such resubmit behavior; timeout is terminal.

**Why it happens:** Mixing adapter-internal retry (tenacity) with engine-level retry (future work) can cause retry amplification or client_order_id mismatches. Phase 02.1 does NOT introduce engine-level retry — keep it that way.

**How to avoid:** Explicitly document in the plan: "Phase 02.1 does NOT add engine-level retry on timeout. Timeout → cancel-recovery → mark order FAILED (or CANCELLED if orphan cancel succeeded) → propagate to `_live_execution` → may trigger one-leg recovery if the OTHER leg filled. Any future engine-level resubmit design must re-examine client_order_id semantics." This boundary note belongs in the 02.1-SUMMARY.md.

**Warning signs:** Retry logic at two layers without explicit coordination on idempotency keys.

```python
# ... inside ExecutionEngine ...

async def _place_order_for_leg(
    self, arb_id, platform, market_id, canonical_id, side, price, qty,
) -> Order:
    adapter = self.adapters.get(platform)
    if adapter is None:
        return Order(
            order_id=f"{arb_id}-{side.upper()}-NOADAPTER",
            platform=platform, market_id=market_id, canonical_id=canonical_id,
            side=side, price=price, quantity=qty,
            status=OrderStatus.FAILED, timestamp=time.time(),
            error=f"No adapter configured for platform: {platform}",
        )

    try:
        order = await asyncio.wait_for(
            adapter.place_fok(arb_id, market_id, canonical_id, side, price, qty),
            timeout=self.execution_timeout_s,
        )
    except asyncio.TimeoutError:
        # CR-01 fix: look up any orders placed with this arb_id prefix, cancel each.
        partial = Order(
            order_id=f"{arb_id}-{side.upper()}-{platform.upper()}",
            platform=platform, market_id=market_id, canonical_id=canonical_id,
            side=side, price=price, quantity=qty,
            status=OrderStatus.PENDING, timestamp=time.time(),
            error=f"local timeout after {self.execution_timeout_s}s",
        )
        cancelled_any = False
        found_count = 0
        prefix = f"{arb_id}-{side.upper()}-"
        try:
            open_orders = await adapter.list_open_orders_by_client_id(prefix)
        except Exception as exc:
            logger.warning(
                "timeout_recovery.lookup_failed platform=%s arb_id=%s err=%s",
                platform, arb_id, exc,
            )
            open_orders = []
        found_count = len(open_orders)
        for real in open_orders:
            try:
                if await adapter.cancel_order(real):
                    cancelled_any = True
            except Exception as cancel_exc:
                logger.warning(
                    "timeout_recovery.cancel_raised platform=%s order_id=%s err=%s",
                    platform, real.order_id, cancel_exc,
                )
        if cancelled_any:
            partial.status = OrderStatus.CANCELLED
            partial.error += f"; cancelled {found_count} orphaned order(s)"
        elif found_count > 0:
            partial.status = OrderStatus.FAILED
            partial.error += f"; found {found_count} orphaned order(s) but cancel failed - manual reconciliation required"
        else:
            partial.status = OrderStatus.FAILED
            partial.error += "; no matching open order found - platform may have rejected or never received"
        order = partial

    # EXEC-02 / D-16 persistence — unchanged except for _derive_client_order_id
    if self.store is not None:
        try:
            client_order_id = self._derive_client_order_id(order)
            await self.store.upsert_order(order, arb_id=arb_id, client_order_id=client_order_id)
        except Exception as exc:
            logger.warning("ExecutionStore upsert_order failed: %s", exc)
    return order


@staticmethod
def _derive_client_order_id(order: Order) -> Optional[str]:
    """Return the adapter-populated client_order_id, or None.

    Kalshi adapter populates Order.external_client_order_id with the
    `ARB-{n}-{SIDE}-{hex}` string used as the Kalshi idempotency key.
    Polymarket has no client_order_id concept and leaves it None.
    """
    return order.external_client_order_id
```

### Known Threat Patterns for Execution Engine Bug Fix

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| **T-02.1-01: Orphaned order leaks unhedged position risk after timeout** | Denial of Service / Integrity (trading safety) | This is the bug CR-01 fixes. Mitigation: the new timeout-recovery path uses `list_open_orders_by_client_id` + `cancel_order` to convert "unknown state" into "platform-confirmed cancelled or FAILED-with-manual-reconciliation-required incident." VERIFIED by test_timeout_cancels_orphaned_orders_found_by_prefix. |
| **T-02.1-02: Timeout-recovery retry storm against Kalshi during API slowness** | Denial of Service (self-inflicted rate-limit ban) | The new code calls `list_open_orders_by_client_id` exactly ONCE per timeout (inside the `except asyncio.TimeoutError` branch). The adapter's internal tenacity wrapping on `_list_orders` retries up to 3 times with exponential backoff (0.5s → 10s). Combined, a single timeout produces at most 3 list-order requests + N cancel requests where N = orphan count (typically 0-1). Adapter's `rate_limiter` (10 writes/sec) and `circuit` (5-failure threshold) further gate. No amplification risk. |
| **T-02.1-03: `client_order_id` prefix collision across concurrent arbs** | Tampering (wrong order cancelled) | `arb_id` is generated with monotonic counter `f"ARB-{self._next_arb_id:06d}"` and unique per arb. The prefix `f"{arb_id}-{side.upper()}-"` is therefore unique per arb-leg. Cross-arb prefix collision is impossible. Within a single arb-leg, only one `place_fok` ever runs, so prefix matches at most one in-flight order. |
| **T-02.1-04: Kalshi returns malformed Order data in `list_open_orders_by_client_id` response** | Tampering (deserialization) | Existing `KalshiAdapter._order_data_to_order` handles malformed fields with try/except around `float`/`int` conversions [VERIFIED: kalshi.py:388-419]. Malformed orders are constructed with zeroed numeric fields, not raised as exceptions. No new parse surface. |
| **T-02.1-05: `external_client_order_id` leaked in logs as secret** | Information Disclosure | The field contains `ARB-000042-YES-deadbeef` — not secret (no API keys, no PII). Safe to log. Existing `_strip_secrets` structlog processor (Phase 2 Plan 01) still runs — it won't redact this field, but the field's non-sensitive nature makes that correct. |

No new trust boundaries introduced. No new network endpoints. No new auth paths. The fix is entirely internal orchestration logic + one dataclass field.
