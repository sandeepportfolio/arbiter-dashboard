<objective>
Wire per-adapter rate limiting at every outbound call site and expose the rate-limiter state to the dashboard via a periodic `rate_limit_state` WebSocket event. The existing `RateLimiter` class (arbiter/utils/retry.py lines 224-332) already has acquire() + apply_retry_after() + stats; this plan just wraps every adapter HTTP/SDK call and adds the periodic broadcaster. No new dependencies.

Purpose: SAFE-04 — the system must not get banned or throttled by venue rate limits. Kalshi caps at 10 writes/sec per member; Polymarket has comparable limits. Without pre-call acquire(), a burst of opportunities hammers the API and triggers 429 → circuit open → trading halts. With this plan, backoff is transparent and visible to the operator.

Output:
- KalshiAdapter and PolymarketAdapter: `await self.rate_limiter.acquire()` as line 1 of every outbound method
- 429 handling: `apply_retry_after` + circuit.record_failure + FAILED order return (no retry for FOK)
- New `_rate_limit_broadcast_loop` task in api.py emitting rate_limit_state every 2 seconds
- `/api/system` snapshot includes `rate_limits` key
- Unit tests for acquire-before-I/O, 429-retry-after, no-retry-on-FOK, WS event shape
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PlatformAdapter → venue API (Kalshi / Polymarket) | Outbound HTTP / SDK; credentials attached; must respect venue rate limit |
| RateLimiter state → WebSocket subscribers | Outbound; read-only broadcast to all WS clients (public + ops) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-04-A | Denial of Service | Rate limiter bypass via parallel adapter sessions | mitigate | RateLimiter instance is per-adapter, shared across all callers of that adapter. Since only ExecutionEngine constructs and uses the adapters, no second session exists. Constructor test: verify `engine.adapters[p].rate_limiter is the single shared instance`. |
| T-3-04-B | Denial of Service | Rate limiter starvation under load | mitigate | Token bucket refill rate configured per SafetyConfig; emergency cancel_all in plan 03-05 will use chunked acquire with per-chunk limiter; not a concern for this plan. |
| T-3-04-C | Denial of Service | Venue bans us by sending 429 repeatedly | mitigate | `apply_retry_after` respects the venue's Retry-After header exactly; FOK orders never retry after 429 (eliminates runaway request spiral); `circuit.record_failure` eventually opens the breaker after threshold. |
| T-3-04-D | Info Disclosure | rate_limit_state broadcast exposes trading-activity-level metrics | accept | Token consumption is not itself sensitive — only reveals that the system is actively trading; acceptable per existing public read-only model. |
| T-3-04-E | Tampering | Forged Retry-After header tricks us into long backoff | mitigate | `apply_retry_after` already caps penalty at a maximum via fallback_delay when header parsing fails; add explicit upper bound of 60 seconds to the adapter's callsite (pass the header through but don't trust unbounded values). Concrete: `delay = min(delay, 60.0)` after apply_retry_after return. |
| T-3-04-F | DoS | Periodic broadcast floods slow WS clients | mitigate | `put_nowait` + QueueFull logger.debug pattern (from plan 03-01) already handles slow consumers; 2s interval is conservative. |

No HIGH-severity threats remain unmitigated.
</threat_model>
