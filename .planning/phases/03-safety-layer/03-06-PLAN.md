<objective>
Extend the market-mapping schema with a resolution-criteria structure that captures each platform's settlement rules side-by-side, surface the data through the existing `/api/market-mappings` endpoint, emit a `mapping_state` WebSocket event on changes, and persist the schema via an idempotent SQL migration. This plan intentionally ships the DATA layer only — plan 03-07 wires the side-by-side comparison UI on top.

Purpose: SAFE-06 — operator must verify that Kalshi and Polymarket resolve the same underlying question identically before approving an auto-trade mapping. Today the schema lacks fields for this; operators rely on tribal knowledge. This plan makes resolution criteria a first-class part of every mapping record.

Output:
- `MarketMappingRecord` in settings.py gets optional `resolution_criteria` dict
- `MarketMapping` dataclass in market_map.py gets `resolution_criteria_json` + `resolution_match_status` fields
- SQL migration (ALTER TABLE) added idempotently to init.sql + applied on startup
- `/api/market-mappings` GET exposes the fields; POST accepts them
- New WebSocket `mapping_state` event fires when criteria change
- Unit + integration tests
- Dashboard JS tolerance branch — no UI render yet
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| POST /api/market-mappings/{id} body → update_market_mapping | Authenticated operator writes free-text resolution rules |
| MarketMapping persistence → Postgres market_mappings table | Internal; parameterized updates |
| mapping_state WS broadcast → dashboard | Outbound read-only data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-06-A | Tampering | Operator injects malicious mapping to trigger incorrect settlement attribution | mitigate | require_auth guard on POST (operator role only); resolution_criteria is operator-provided data — trust boundary is the authenticated operator; any mis-entry is recoverable via subsequent update (mappings are not settlement ground-truth — the audit pipeline reconciles against real venue data). Operator-provided rules are informational, not authoritative. |
| T-3-06-B | Input Validation | criteria_match accepts arbitrary string → UI downstream bug | mitigate | Explicit enum validation in handle_market_mapping_action: reject any criteria_match not in {"identical","similar","divergent","pending_operator_review"} with 400. |
| T-3-06-C | XSS | Dashboard renders operator-entered rule text | mitigate | Plan 03-07 UI renderer MUST use textContent or escaped innerHTML — noted in plan 03-07 checklist. This plan 03-06 does NOT render, so the XSS surface stays dormant until 03-07. Carry forward as explicit constraint. |
| T-3-06-D | Info Disclosure | Operator free-text note broadcast to unauthenticated WS clients | accept | Resolution rules are public market info (they come from public Kalshi / Polymarket pages). Operator notes might reveal strategy — acceptable for small-capital phase; documented in spec. |
| T-3-06-E | Tampering | SQL injection via resolution_criteria JSONB column | mitigate | All INSERT/UPDATE paths use asyncpg parameterized `$N::jsonb` casts; JSON payload is serialized via `json.dumps` before binding. |
| T-3-06-F | DoS | Malformed JSON in resolution_criteria field crashes serializer | mitigate | `.to_dict()` wraps `json.loads` in try/except → None on failure; API validation rejects non-dict payloads pre-persistence. |

No HIGH-severity threats remain unmitigated for this plan. T-3-06-C is forwarded as a constraint on plan 03-07.
</threat_model>
