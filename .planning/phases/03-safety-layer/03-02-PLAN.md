<objective>
Tighten `RiskManager.check_trade` to enforce per-platform exposure limits IN ADDITION to the existing per-market limit. Emit a structured `order_rejected` incident on every rejection so the operator can see (via existing incident panel) which orders are being blocked and why. Keep the incident event_type distinct so the dashboard can later add a dedicated "Rejected orders" list (plan 03-07) without a new WebSocket event type.

Purpose: SAFE-02 currently checks per-market only; a crypto-correlated trade can pile up $900 on Kalshi (3 × $300 markets) without tripping any alarm. This plan adds per-platform aggregation and surfaces every rejection as a structured incident so operators see safety decisions in real time.

Output:
- `RiskManager` tracks `_platform_exposures: Dict[str, float]` alongside `_open_positions`
- `record_trade` / `release_trade` split exposure across yes/no platforms
- `check_trade` adds per-platform check BEFORE per-market check (fails faster on most-common case)
- `execute_opportunity` calls new `_emit_rejection_incident` helper that records an `order_rejected` incident whenever `check_trade` returns False (not when dedup signature blocks — that's different)
- Unit tests prove all three rejection paths (per-market, per-platform, per-order-rejected-emission)
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-02-A | Tampering | TOCTOU between check_trade and record_trade | mitigate | check_trade and record_trade run within the same `await execute_opportunity` coroutine under asyncio single-thread; no race exists within the process. | T-3-02-B | Tampering | Signed int overflow in exposure accumulation | mitigate | All exposure math uses Python float; `suggested_qty * price` is bounded by ScannerConfig.max_position_usd (hard-capped at scanner level) multiplied by per-market cap.
