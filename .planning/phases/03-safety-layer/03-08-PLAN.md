<objective>
Close the SAFE-02 live-mode gap reported in `.planning/phases/03-safety-layer/03-VERIFICATION.md`: in `_live_execution`, `record_trade` is currently guarded by `if status == "filled":` inside the outer `if status in {"submitted", "filled"}:` block (engine.py:814-824). The outer condition is correct; the inner guard is too strict. Per-platform exposure does not accumulate while orders are in-flight, so a burst of `submitted` orders on the same platform can collectively exceed `SafetyConfig.max_platform_exposure_usd` because the next `check_trade` sees stale exposure data.

**Chosen approach: Option A (record on submitted) with explicit symmetry handling on recovery.**

Why Option A over Option B (reservation model with `reserve_exposure` / `release_exposure`):

1. **Lower blast radius.** Option A is a 1-line guard removal in `_live_execution` (line 815), plus a small surgical edit to handle the `"recovering"` status (record only the surviving leg) and a single `release_trade` call inside `_recover_one_leg_risk` after a successful cancel. Option B would require new public `RiskManager.reserve_exposure` / `release_exposure` methods, modifications to `_place_order_for_leg` (the per-leg dispatcher, called twice from `_live_execution` and several times from tests), and rewiring every test that uses `_place_order_for_leg` directly. The change surface is roughly 3× larger.

2. **Mirrors the dry-run path verbatim.** `_simulate_execution` at engine.py:759-767 already calls `record_trade(opp.canonical_id, total, pnl, yes_platform=..., no_platform=..., yes_exposure=..., no_exposure=...)` unconditionally. Option A makes the live path use the IDENTICAL call shape, which is the dry-run/live parity invariant the verification report explicitly cites. Option B would diverge the live path from the dry-run path with a new abstraction that dry-run doesn't use.

3. **Symmetric within the current one-shot lifecycle.** `_live_execution` returns once with a final status. There is no submitted→cancelled state machine separate from this return — recovery happens INSIDE this same coroutine via `_recover_one_leg_risk` before the function returns. So we have full information at the record-trade decision point: `"failed"` (no exposure → no record), `"recovering"` (one leg's worth of exposure → record only the surviving leg), `"submitted"` / `"filled"` (both legs' exposure → record both). The leakage scenario the verifier worried about ("submitted later resolves to cancelled") is handled inline because `_recover_one_leg_risk` is called BEFORE the record-trade decision: by the time `status == "recovering"` is set on line 793 or 798, the surviving-leg cancellation has already been attempted.

4. **Test additionality is small.** Three focused tests can prove all three observable truths (burst rejection, recovering-only-surviving, dry-run parity) plus a release-on-recovery test. No need for new test infrastructure — the existing `_make_safety_opp` helper, `MagicMock` adapter pattern (test_engine.py:464-481), and `subscribe_incidents()` queue all work.

Output:
- Surgical edit to `_live_execution` (engine.py:814-824): remove the inner `if status == "filled":` guard so both `"submitted"` and `"filled"` record. Add a new branch for `"recovering"` that records only the surviving leg's exposure split.
- Surgical edit to `_recover_one_leg_risk` (engine.py:1045-1049): after `_cancel_order` returns True for a previously-submitted leg, call `release_trade` with that leg's exposure split so the per-platform reservation is freed.
- 4 new tests in `arbiter/execution/test_engine.py` proving: (1) burst of two submitted on same platform — second rejected; (2) recovering status records only surviving leg; (3) recovering→cancellation path releases the reservation; (4) dry-run path unchanged (parity).
- Full test suite stays green: ≥113 passing across `arbiter/execution/` (current baseline per VERIFICATION.md row "Adapter tests"); engine tests stay at ≥25 passing per the 03-02-SUMMARY baseline.
