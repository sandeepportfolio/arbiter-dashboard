# Phase 04 Plan 02.1: KalshiAdapter.place_resting_limit (scope-expansion) Summary

**Added `KalshiAdapter.place_resting_limit` so SAFE-01 kill-switch live-fire can place a true resting Kalshi order; PHASE4_MAX_ORDER_USD hard-lock + rate-limiter + circuit-breaker all mirror `place_fok` exactly, with the only functional differences being (a) `time_in_force` omitted from the order body and (b) `status="resting"` is the expected steady state.**

## Motivation (Scope Expansion Discovery)

During Phase 4 Wave 2 execution, three independent agents (04-03, 04-06, 04-07) flagged the same gap in parallel: `KalshiAdapter` exposed only `place_fok` (fill-or-kill) and no public non-FOK placement method. 04-05 (SAFE-01 kill-switch live-fire) explicitly requires a **resting** order that survives >=5 seconds — the window in which the kill-switch is expected to trip and cancel open orders mid-life. A FOK order fills or dies instantly, so it cannot exercise the kill-switch against a real resting position.

Downstream consumers were each reaching for a raw-HTTP workaround (constructing `session.post` directly using the adapter's private `session+auth` attributes) — a recipe for drift, since any future change to auth headers, rate-limiter plumbing, or PHASE4 hard-lock policy in `place_fok` would silently bypass those workarounds. Adding the method at the adapter layer removes all three workarounds at once.

## Implementation Details

### Method signature

```python
async def place_resting_limit(
    self,
    arb_id: str,
    market_id: str,
    canonical_id: str,
    side: str,            # "yes" | "no"
    price: float,         # (0, 1) exclusive, dollar convention
    qty: int,
) -> Order
```

Mirrors `place_fok` signature verbatim so the engine's `_place_order_for_leg` dispatch shape does not need to change when it eventually routes a leg through `place_resting_limit` (post-Phase 5).

### Gate ordering (top-to-bottom)

1. **Auth gate** — `self.auth.is_authenticated` else FAILED
2. **Price validation** — `0 < price < 1` else FAILED (no HTTP call)
3. **PHASE4_MAX_ORDER_USD hard-lock** — `notional_usd = qty * price > cap` else FAILED (no HTTP call). Unparseable env → `0.0` cap → maximum restriction (safe-default mode).
4. **Circuit-breaker gate** — `circuit.can_execute()` else FAILED (no HTTP call)
5. **Build body** — omit `time_in_force`, include `yes_price_dollars` XOR `no_price_dollars`
6. **Rate-limiter acquire → HTTP POST** (via shared `_post_order` helper — transient-retry aware)
7. **Response handling** — 429 applies Retry-After + FAILED (no retry); non-2xx → circuit failure + FAILED; 2xx → `_RESTING_STATUS_MAP` lookup (resting → SUBMITTED without warning)

The hard-lock deliberately fires **before** the circuit-breaker check so a cap breach gets its dedicated rejection reason even if the circuit is also open — operators should see the true root cause (over-cap notional), not a "circuit open" masking message.

### Status mapping

A new `_RESTING_STATUS_MAP` dict, initialized as a copy of `_FOK_STATUS_MAP`, carries the same `"executed"/"canceled"/"pending"/"resting"` mappings but is consulted from `place_resting_limit` specifically. The semantic difference: in `place_fok`, `"resting"` triggers a `kalshi.fok.unexpected_resting` warning because FOK should never rest; in `place_resting_limit`, `"resting"` is the expected happy-path terminal and no warning is emitted.

### Shared helper reuse

Both `place_fok` and `place_resting_limit` call the same `_post_order(body)` helper (tenacity-decorated with `@transient_retry()` and rate-limiter acquire inside). This means:

- Any future change to retry policy, rate-limit acquisition, or header construction automatically applies to both placement variants.
- Kalshi's `client_order_id` deduplication covers both methods identically.

## Deviations from Plan

**1. [Signature — Rule 2: consistency with existing adapter convention] Used `price: float` (0-1 dollar convention) rather than plan-suggested `price_cents: int` (1-99)**
- **Found during:** Task 1 test scaffolding (noticed the existing `place_fok` uses `price: float` in dollars).
- **Issue:** The inline plan proposed `price_cents: int` based on Kalshi's raw API using `yes_price`/`no_price` cent-integer fields. However, the existing `place_fok` in this same adapter takes `price: float` in (0,1) and serializes it as `yes_price_dollars`/`no_price_dollars` at the wire — which is a separate, documented Kalshi API field that accepts the dollar format. The engine's `_place_order_for_leg` dispatches using `opp.yes_price` / `opp.no_price` which are float dollars from `ArbitrageOpportunity`. Making `place_resting_limit` take cents would require a cents-vs-dollars translation layer somewhere above the adapter.
- **Fix:** Kept the `price: float` signature with the dollar convention, matching `place_fok`. The wire body uses `yes_price_dollars` / `no_price_dollars` (Kalshi's documented dollar fields) — identical to the place_fok body shape except for the omitted `time_in_force`.

## Interface Contract Published

```python
# EXEC-04 Protocol extension (additive; NOT breaking):
class KalshiAdapter:
    async def place_resting_limit(
        self,
        arb_id: str,
        market_id: str,
        canonical_id: str,
        side: str,             # "yes" | "no"
        price: float,          # (0, 1) exclusive
        qty: int,
    ) -> Order: ...
```

Return-value contract:
- **Happy path (Kalshi accepts):** `Order(status=OrderStatus.SUBMITTED, order_id=<real Kalshi id>, external_client_order_id=<ARB-...-YES-hex>, fill_qty=0, fill_price=<echoed>, ...)`. `cancel_order(order)` works without translation.
- **Hard-lock breach:** `Order(status=OrderStatus.FAILED, error="PHASE4_MAX_ORDER_USD hard-lock: ...")`.
- **Auth / invalid price / circuit open:** `Order(status=OrderStatus.FAILED, error=<specific reason>)`.
- **429 from Kalshi:** `Order(status=OrderStatus.FAILED, error="rate_limited (X.Xs)")` — NO retry.
- **5xx / network error:** `Order(status=OrderStatus.FAILED, error=<Kalshi API status: body>)` — circuit failure recorded.
